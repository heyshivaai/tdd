ANALYSIS RULES:
1. SYNTHESIZE, DON'T REPEAT. If three signals all point to "weak security posture," produce ONE finding that ties them together, not three separate findings restating each signal.
2. CONTRADICTIONS ARE GOLD. If document A says one thing and document B says another, that's a finding. Call it out explicitly with both sources.
3. ABSENCE IS EVIDENCE. If the VDR should contain something for this domain but doesn't (e.g., no DR plan for a healthcare company), that's a finding, not just a blind spot.
4. QUANTIFY WHEN POSSIBLE. "Tech debt is high" is useless. "The codebase has 340K lines with no test coverage mentioned in any document" is actionable.
5. EVERY FINDING NEEDS EVIDENCE. Do not produce findings without at least one evidence item (signal reference, document excerpt, or documented absence).
6. GRADE HONESTLY. RED means "this domain has issues that could affect deal value or timeline." YELLOW means "concerns exist but are manageable." GREEN means "this domain is solid."
7. QUESTIONS MUST BE SPECIFIC. Not "tell us about your security practices" but "provide the most recent SOC2 Type II audit report and any open remediation items from that audit."
8. WRITE FOR A PE PARTNER. They are smart but not technical. Translate everything to business impact — cost, time, risk, deal value.
9. EVIDENCE EXCERPTS MUST BE VERBATIM. Copy exact text from the documents — do not paraphrase evidence quotes.
10. SORT FINDINGS BY SEVERITY. CRITICAL first, then HIGH, MEDIUM, LOW.

DOMAIN-SPECIFIC DEEP ANALYSIS (v1.4 calibration-enhanced — apply when relevant to this pillar):

FOR OrganizationTalent:
- Assess org structure: Is there a single tech leader (CTO/CIO) or fragmented leadership? Count CEO direct reports with tech responsibility.
- Identify consolidation opportunities: Could product, engineering, data, and security be unified under one leader?
- Evaluate FTE vs contractor dependency: What % of engineering is offshore/contractor? Is core IP owned by contractors?
- Produce a restructuring recommendation if warranted (e.g., "consolidate under CTO, reduce CEO direct reports from 7 to 4").

FOR RDSpendAssessment:
- Calculate R&D as % of revenue when both figures are available. Benchmark: SaaS growth-stage 15-25%, mature 10-15%.
- Itemize hosting costs by provider (AWS, GCP, Azure, on-prem). Calculate annualized totals from monthly data.
- Flag customer concentration: any customer >10% of CARR, especially on legacy platform.
- Assess cost scaling: does hosting cost grow linearly with customers (bad) or sub-linearly (good)?
- Provide total R&D spend figure and composition (FTE labor, contractor labor, tools, hosting).

FOR TechnologyArchitecture:
- Itemize tech debt: list each specific item (legacy framework, deprecated API, etc.) with estimated cost and timeline if available.
- Assess multi-cloud risk: count environments, migration dependencies (customer interfaces, integrations), and timeline realism.
- Extract tooling landscape: development tools, CI/CD, testing, monitoring, security tools. Flag gaps.
- Evaluate platform consolidation opportunity with specific product-to-platform mapping.

FOR SecurityCompliance:
- When compliance certifications exist (SOC2, HITRUST, HIPAA), start from GREEN baseline and downgrade only for material gaps.
- Active data incidents (PHI breaches, PII leaks) override certification status — these are CRITICAL regardless.
- Assess incident detection capability: MTTD >7 days = concern, >30 days = critical.
- Evaluate vendor security management: one-time review at onboarding vs annual re-assessment.

FOR SDLCProductManagement:
- Extract product portfolio details: number of products, technology platform per product, customer retention rate.
- Assess product-market fit signals: retention >95% = strong, 90-95% = good, <90% = concern.
- Identify platform fragmentation: multiple products on different stacks = consolidation opportunity.

FOR InfrastructureDeployment:
- Extract hosting cost breakdown by provider with monthly trend data.
- Assess DR readiness: formal RTO/RPO vs informal resilience. Note: tested failover with no formal doc = YELLOW, not RED.
- Evaluate monitoring maturity: which tools, coverage gaps, alerting quality.

REMEDIATION COST ESTIMATION:
For every finding with severity CRITICAL or HIGH, provide a remediation cost estimate in the remediation block. Use these heuristics when specific costs aren't in the documents:
- Process/policy creation: $5K-$25K (1-2 months, 1 person)
- Tool implementation: $25K-$100K (1-3 months, team of 2-3)
- Legacy migration (small component): $50K-$200K (3-6 months)
- Legacy migration (major platform): $200K-$1M (6-18 months)
- Security remediation (OSS/vulnerability): $100K-$150K (1-3 months, offshore team)
- Org restructuring: minimal direct cost, 3-6 month execution
When the documents provide actual estimates, use those instead of heuristics.
